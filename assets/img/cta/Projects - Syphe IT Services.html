Docker

Docker Build, Push and Pull

The Container Journey

From Docker website image on Local laptop to Master Node Source Code to a Running website Pod in OCI

01 Docker Build: The Blueprint

When you run docker build, you are following a "recipe" called a Dockerfile. Docker reads this file from bottom to top, creating read-only layers.

  • Base Image: Usually a lightweight OS like alpine or debian-slim.
  • The Tag: You must tag the image with the registry URL (e.g., ord.ocir.io/...) so Docker knows where its "home" is.
  • DockerFile:

    FROM nginx:stable-alpine

    COPY . /usr/share/nginx/html

    EXPOSE 80

# Example Build Command
docker build -t ord.ocir.io/tenancy/syphe-web:v1 .

02 The Registry: Push & Pull

Docker images are stored in a **Registry** (like OCIR). Think of this as GitHub, but for compiled containers instead of code.

  • Pushing: Uploads your local layers to the cloud. Only *new* layers are uploaded, saving bandwidth.
  • Pulling: Downloading the image. You usually do this manually only for testing; Kubernetes handles this automatically during deployment.
docker push ord.ocir.io/tenancy/syphe-web:v1

03 How K8s Gets the Image

This is the "handshake" between the Master and the Worker. Here is exactly what happens when you run kubectl apply:

1. The Instruction

The iMac (Master) sends the Deployment YAML to the OCI Worker. It says: "Run syphe-web:v1."

2. The Authentication

The Worker looks for an ImagePullSecret. This contains your Oracle Auth Token. Without it, the Worker is denied entry to OCIR.

3. The Execution

The Worker's Container Runtime (CRI-O or Containerd) downloads the layers from OCIR and starts the Pod.

Critical: The ImagePullPolicy

If you keep the same tag (like :latest) and update your code, Kubernetes might think it already has the image and won't download the new version. To fix this, always include this in your YAML:

spec:
  containers:
  - name: syphe-app
    image: ord.ocir.io/tenancy/syphe-web:latest
    imagePullPolicy: Always # This forces the Worker to check the Registry every time
Does K8s actually build, pull, and push?

The short answer is no. Kubernetes does not build or push images.

Think of it this way: Docker is the "Manufacturer" (Build & Ship), while Kubernetes is the "Warehouse Manager" (Pull & Organize).

1. The Build and Push Phase (Docker's Job)

Kubernetes has no native way to look at your source code and turn it into a container.

  • Building: You use Docker on your local iMac or laptop to run docker build. This creates the "Shipping Container."
  • Pushing: You use Docker to run docker push. This sends that container to a Registry.
  • K8s Role: None. Kubernetes doesn't even know your code exists until the image is already sitting in a registry.
2. The Pull and Run Phase (Kubernetes' Job)

This is where Kubernetes takes over, but it doesn't use "Docker Desktop." It uses something called a Container Runtime (like containerd).

  • Pulling: When you tell K8s to deploy your app, the Kubelet (the agent on your OCI worker node) says: "I need to run syphe-web:v1. I see it's not here, so I will download it from the registry."
  • Running: Once pulled, K8s starts the container. It doesn't "push" it anywhere; it just keeps it running and restarts it if it crashes.
Why the confusion? (The "Docker-shim" history)

In the old days, Kubernetes actually used the Docker software itself to pull and run containers. However, modern Kubernetes has moved away from this. It now uses a "lighter" version of the same technology called containerd.

The separation is now very clear:
  1. You use Docker to Create the image.
  2. The Registry (like Docker Hub or OCIR) Stores the image.
  3. Kubernetes uses its runtime to Pull and Run the image.
Action Done by Docker? Done by Kubernetes?
Build Image Yes (on your laptop/iMac) No
Push Image Yes (to the cloud) No
Pull Image Yes (if manual) Yes (automated by the Worker Node)
Run Container Yes (locally) Yes (in the cluster)
